Event Listeners

Events Listeners are best used for actions that can happen multiple times on the same element, such as:

  • click
  • touchstart
  • submit
  • on [data,close,...]
  • etc.

With these type of events you usually do not need to worry about the state before you attached the listener. Events may never trigger.

Callbacks

Callbacks are the most common argument for event listeners or asynchronous functions.
The functions take some input and a callback that's invoked with some kind of output. 


            function getUsernameWithId(id, cb) {
                var username = '';
                var error = null;
                ...
                cb(error, username);
            }
                

            getUsernameWithId(someId, function (err, value) {
                // in callback, we have a error and value from function
            });

Promises

By removing the imperative nature of a callback, we can proxy the value needed for later use. 


                function getUsernameWithId(id) {
                    return new Promise(function (resolve, reject) {
                        var username = '';
                        ...
                        if(username)
                            resolve(username);
                        else
                            reject('user not found!');
                    });
                }

Promise me this


        getUsernameWithId(someId)
            .then(function (value) {
                // we receive value here
            })
            .catch(function (error) {
                // Handle any error thrown, handle rejections
            });

The most notable improvement of using a promise instead of just callbacks is that you have an instance of an object that represents deferred work.


              var usernamePromise = getUsernameWithId(someId);

Promises vs Callbacks

Seemingly the same, but conceptually different

  • Promises are:
    • functional programming
    • a form of declarative programming
    • concerned with the WHAT
  • Callbacks are:
    • procedural programming
    • a form of imperative programming
    • concerned with the HOW

If I write my little library, and it goes and talks to Redis, and that’s the last thing it ever does, I can just pass the callback that was handed to me off to Redis. And when we do hit these problems like callback hell, I’ll tell you a secret: there’s also a coroutine hell and a monad hell and a hell for any abstraction you create if you use it enough.

For the 90% case we have this super super simple interface, so when you need to do one thing, you just get one little indent and then you’re done. And when you have a complicated use case you go and install async like the other 827 modules that depend on it in npm.

Mikeal Rogers, LXJS 2012

We are promises, and we're here to help

When it comes to asynchronous requests or operations, tracking all successes and failures can become cumbersome.

Example

The infamous pyramid of doom.


             step1(function (value1) {
                 step2(value1, function(value2) {
                     step3(value2, function(value3) {
                         step4(value3, function(value4) {
                            // Do something with value4
                         });
                     });
                 });
             });
source: https://github.com/kriskowal/q

Example

The more manageable tower of do awesome.


    var step1 = new Promise(...);

    step1.then(step2)
        .then(step3)
        .then(step4)
        .then(function (value4) {
            // Do something with value4
        })
        .catch(function (error) {
            // Handle any error from any above steps
            throw new Error('fatal error in promises:' + error.message);
        });

Think of a promise as:

a promise to eventually having a value at some indeterminable point in the future.

It can be used for making multiple potentially long tasks operate within a synchronous set of instructions.

Promise Analogy: Part 1

Every morning, Samwise asks Hodor to go out and collect food from the traps.

Hodor promises Samwise that he will go out and collect the trappings.

Hodor.

Then Samwise decides that if:

  1. Hodor finds rabbit, they'll have rabbit stew with some potatoes.
  2. Hodor finds pheasant, they'll eat roast pheasant.
  3. Hodor finds no trappings, they'll eat Lembas bread.
The promise is now pending

Promise Analogy: Part 2

Soon Hodor returns.

Different outcomes may have occurred.

  1. Promise fulfilled: Hodor kept his promise and went to the trap, and found rabbit. Rabbit stew it is!
  2. Promise fulfilled: Hodor kept his promise and went to the trap, and found pheasant. Time to roast!
  3. Promise fulfilled: Hodor kept his promise and went to the trap, and found nothing. Time to eat bread!
  4. Promise rejected: Hodor went out to collect the traps but saw White Walkers and Nazgûl - it was too dangerous to venture out. Time to eat bread!
  5. Promise rejected: Hodor. Time to eat bread!
